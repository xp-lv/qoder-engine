# 死链与孤儿检测规则

## 1. 死链（Dead Link）

**定义**：某角色声明了某个 input 文件，但该文件在整个 app 中没有任何角色的 output，也不在 knowledge 的 inject_to 中，也不是外部文件（如 00-需求描述.md）。

**检测逻辑**：
```
对于每个角色 R 的每个 input 文件 F：
  如果 F 不在 knowledge/ 路径下：
    如果 F 不在 00- 开头（外部文件）：
      如果没有任何角色的 output 包含 F：
        → F 是死链
```

**严重度**：
- `high`: 死链文件是角色的唯一 input（角色完全无法执行）
- `medium`: 死链文件是多个 input 之一（角色部分缺失）

## 2. 孤儿物料（Orphan Material）

**定义**：某角色声明了某个 output 文件，但该文件在整个 app 中没有任何其他角色的 input 引用它。

**检测逻辑**：
```
对于每个角色 R 的每个 output 文件 F：
  如果没有任何其他角色的 input 包含 F：
    如果 F 不在终审/最终产物中（如 终审裁决书、审计摘要）：
      → F 是孤儿物料
```

**例外**：最终产物（终审裁决书、架构蓝图等）允许是孤儿，因为它们是 app 的最终输出。

**严重度**：
- `low`: 孤儿是最终产物（合理）
- `medium`: 孤儿是中间产物但无人消费（可疑）
- `high`: 孤儿是 process/outputs/ 下的文件且无人消费（浪费计算）

## 3. 断路角色（Broken Chain）

**定义**：在 DAG 拓扑中，角色与角色之间的路由边不连通。

**检测逻辑**：
```
1. 构建角色 DAG（基于 edges）
2. 检查是否存在从入口不可达的角色
3. 检查是否存在无法到达任何终态的角色
```

**类型**：
- `unreachable`: 角色无入边且不是入口角色
- `no_exit`: 角色无出边且不是终态角色
- `disconnected`: 角色入边和出边都存在，但在 DAG 中形成了孤岛

## 4. 物料一致性

**定义**：边 carries 声明携带的文件，目标角色的 inputs 中是否声明了对应该文件。

**检测逻辑**：
```
对于每条 carries 文件 C：
  检查目标角色的 inputs 中是否有 C
  如果没有 → carries 可能无效（目标角色未声明读取）
```

**严重度**：
- `low`: carries 文件在目标 inputs 中未声明（可能依赖运行时注入）
- `info`: 仅提示，不判为问题

## 5. 检查输出格式

所有检测结果写入 JSON，统一 schema：

```json
{
  "result": {
    "verdict": "confirmed | issues_found",
    "dead_links": [
      {
        "role": "角色名",
        "file": "文件路径",
        "severity": "high | medium",
        "reason": "没有任何角色产出该文件"
      }
    ],
    "orphan_materials": [
      {
        "role": "角色名",
        "file": "文件路径",
        "severity": "low | medium | high",
        "reason": "无其他角色消费该文件"
      }
    ],
    "broken_chains": [
      {
        "role": "角色名",
        "type": "unreachable | no_exit | disconnected",
        "severity": "high",
        "reason": "..."
      }
    ]
  }
}
```

## 6. 正反例

### 死链正例（检测到死链）
```yaml
# app.yaml 片段
roles:
  角色C:
    inputs:
      - 中间产物: process/outputs/中间.json  # 无任何角色产出此文件
```
检测结果：`{"role": "角色C", "file": "process/outputs/中间.json", "severity": "high", "reason": "无任何角色产出该文件"}`
原因：`process/outputs/中间.json` 不是 knowledge/，不是 external，且无 producer。

### 死链反例（不是死链）
```yaml
roles:
  角色D:
    inputs:
      - 知识文件: knowledge/规范.md     # knowledge，允许无 producer
      - 入口文件: 00-需求.md           # external，允许无 producer
```
检测结果：无 finding。原因：knowledge/ 和 00- 开头的文件天然不需要 producer。

### 孤儿正例（合理的孤儿）
```yaml
roles:
  终审裁决者:
    outputs:
      - 裁决书: outputs/裁决书.json  # 最终产物，无人消费但合理
```
检测结果：`{"role": "终审裁决者", "file": "outputs/裁决书.json", "severity": "low", "reason": "最终产物，无人消费但合理"}`
原因：最终产物允许孤儿（low 级，不需要修复）。
