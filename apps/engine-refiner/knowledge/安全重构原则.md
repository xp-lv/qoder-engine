# 安全重构原则

> 本文档定义行为保持型重构的安全操作规范，供修复方案设计师设计修复方案。

---

## 核心铁律：行为不变性

**所有代码改动必须保持外部行为完全等价。**

"外部行为"指：
- 函数的输入→输出映射不变
- 函数的副作用（文件写入、状态修改、异常抛出）不变
- 函数的调用方式（签名、返回值结构）不变

---

## 安全重构操作清单

### ✅ 安全操作（保持行为等价）

| 操作 | 定义 | 安全条件 |
|---|---|---|
| **提取方法** | 将函数内一段代码提取为独立内部函数 | 提取的代码块无局部变量逃逸；新函数仅被原函数调用 |
| **内联变量** | 将只赋值一次的变量替换为其值 | 变量确实只被赋值一次且无外部引用 |
| **重命名** | 修改变量/函数/参数名 | 无同名冲突；所有引用点同步更新 |
| **简化条件** | 合并重复条件判断、消除冗余分支 | 逻辑真值表等价 |
| **删除死代码** | 移除从未执行的代码 | 有静态证据证明不可达（如 version 标记已废弃） |
| **提取常量** | 将魔法数字/字符串提取为命名常量 | 常量值与原值完全相同 |
| **统一重复模式** | 将多处复制的相同逻辑抽取为公共函数 | 公共函数行为与各处复制版本完全一致 |
| **添加类型标注** | 为无类型标注的函数添加类型提示 | 不改变运行时行为（type hint 不影响执行） |
| **条件日志开关** | 将硬编码日志改为环境变量控制 | 默认行为不变（环境变量未设置时保持原行为） |

### ❌ 危险操作（可能改变行为）

| 操作 | 风险 |
|---|---|
| 修改函数签名 | 调用方不兼容 |
| 改变控制流语义 | if→while、return→break 等改变执行路径 |
| 修改数据结构 | dict→list、list→set 改变迭代顺序 |
| 修改异常类型 | try/except 不再捕获正确的异常 |
| 引入新依赖 | import 新模块改变命名空间 |
| 修改 import 方式 | subprocess→function import 可能引入递归/死锁 |

## 方案设计规范

每个修复方案必须包含以下字段：

```json
{
  "id": "C001",
  "target_finding": "F001",
  "file": "engine/scripts/state_io.py",
  "lines": "42-51",
  "operation_type": "提取方法",
  "description": "将诊断日志提取为独立函数 _diag_log()，用环境变量 ENGINE_DIAG_LOG 控制开关",
  "behavior_preserved": true,
  "equivalence_reason": "日志写入不影响 STATE.json 内容；提取为函数后执行顺序和写入内容完全不变；环境变量未设置时 _diag_log 直接 return，等价于注释掉的效果",
  "elegance_score": 8,
  "elegance_reason": "将生产路径中的调试逻辑隔离到独立函数，通过环境变量控制开关，符合关注点分离原则",
  "risk_assessment": "low",
  "risk_detail": "唯一风险：环境变量名拼写错误导致日志不输出，但日志仅为诊断用途不影响功能",
  "diff_preview": "..."
}
```

### elegance_score 评分标准

| 分数 | 标准 |
|---|---|
| 9-10 | 机制级修复：从根因消除问题，提升架构层次 |
| 7-8 | 优雅修复：隔离问题、提升可维护性，无明显副作用 |
| 5-6 | 合格修复：解决了问题但方案有改进空间 |
| 3-4 | 补丁修复：治标不治本，引入了新的复杂度 |
| 1-2 | 劣质修复：打补丁，引入更多问题 |

### risk_assessment 判定标准

| 级别 | 标准 |
|---|---|
| low | 改动隔离在单一函数内，无外部调用方受影响 |
| medium | 改动涉及多个函数或文件，需要同步更新引用 |
| high | 改动影响公共 API 或核心数据流（不应出现在本 APP 中） |

---

## 行为等价性验证方法

设计方案时需预填以下验证策略：

1. **AST 结构对比**：提取修改前后的 AST，对比关键逻辑节点
2. **接口扫描**：确认所有公开函数的签名、返回值结构未变
3. **导入链检查**：确认无新增/删除的 import 依赖
4. **Dry-run 回归**：用测试输入运行修改前后的脚本，对比 stdout
