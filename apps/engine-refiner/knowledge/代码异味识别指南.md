# 代码异味识别指南

> 本文档定义 4 个维度的代码检查项标准，供漏洞发现者系统性扫描引擎代码。

---

## 维度一：效率

查找不必要的性能开销和资源浪费。

| 检查项 | 描述 | 示例 |
|---|---|---|
| E1-生产路径诊断日志 | 生产代码路径中嵌入调试/诊断日志写入（I/O 开销） | state_io.py 每次 `_write_unlocked` 都执行 traceback 提取 + 文件追加写入 |
| E2-重复计算 | 同一函数内对同一表达式重复计算 | 同一值在循环内多次计算 `len(state.get(...))` |
| E3-冗余文件读取 | 同一文件在同一调用链中被多次完整读取 | orchestrator 在一次 dispatch 中多次读 ROUTER.json |
| E4-不必要的序列化 | 对大型对象做 JSON 序列化但只用到少数字段 | 只需要 `steps` 却 json.load 整个 ROUTER |
| E5-同步阻塞等待 | 非必要的同步等待（可以异步或跳过的场景） | subprocess.run 无 timeout 或 timeout 过长 |

## 维度二：简洁

查找可以删除或简化的冗余代码。

| 检查项 | 描述 | 示例 |
|---|---|---|
| S1-死代码 | 定义但从未被调用的函数/变量/导入 | import 了 uuid 但从不使用 |
| S2-注释掉的代码 | 被注释掉但保留在文件中的旧实现 | `# v4.x 旧逻辑...` 大段注释 |
| S3-冗余版本标记 | 版本注释散落且不统一（v9.1/v9.2/v4.2 混用） | 同一文件中同一函数有 v9.1 和 v9.2 两个版本注释 |
| S4-过度防御 | 对不可能为 None 的值做多层 None 检查 | 已在函数入口检查过的值在循环内重复检查 |
| S5-重复模式 | 相同的代码模式在多处复制粘贴 | output() / output_error() 在 3+ 文件中重复定义 |

## 维度三：一致性

查找命名、风格、模式的不统一。

| 检查项 | 描述 | 示例 |
|---|---|---|
| C1-函数命名不一致 | 同类功能使用不同命名风格 | load_json vs load_state vs _io_load |
| C2-错误处理不统一 | 同类错误在不同文件中用不同方式处理 | 有的用 output_error()，有的用 raise，有的用 sys.exit(1) |
| C3-重复函数名 | 不同文件中同名函数做相似但不完全相同的事 | output() 在 router/orchestrator/step 各有一份 |
| C4-魔法字符串 | 硬编码的状态值/错误码/字段名 | "step_status" / "terminal_state" 散落硬编码 |
| C5-导入风格混合 | 有的用 sys.path.insert，有的用相对导入 | |

## 维度四：优雅

查找结构性设计问题。

| 检查项 | 描述 | 阈值 |
|---|---|---|
| Y1-过长函数 | 单个函数体超过阈值行数 | > 80 行 |
| Y2-深层嵌套 | 条件/循环嵌套屽数过深 | > 4 层 |
| Y3-过多参数 | 函数参数过多 | > 5 个 |
| Y4-职责混合 | 一个函数/类做太多事情 | router.py main() 兼做解析+路由+JOIN+完成判定 |
| Y5-魔法数字 | 代码中出现的未命名常量 | timeout=30, max=3 等 |
| Y6-缺少类型标注 | 函数签名缺少类型提示 | 公开函数无类型标注 |

---

## 使用方法

1. 对每个引擎文件逐行扫描，按 4 维度检查项逐项匹配
2. 对每个发现的 issue 记录：文件路径、行号范围、维度编号、严重度（high/medium/low）、描述
3. 严重度判定标准：
   - **high**：影响生产路径性能 / 存在 bug 风险 / 架构级问题
   - **medium**：可维护性受影响 / 存在重复代码
   - **low**：风格问题 / 命名不一致
4. 产出漏洞清单 JSON，每条 finding 包含 id、file、lines、dimension、severity、description、suggestion

## 高频检查项正反例对照

### E1-生产路径诊断日志

❌ **问题代码**：
```python
def _write_unlocked(state_path, state):
    import traceback as _tb, time as _time  # 生产路径中导入调试模块
    _caller = _tb.extract_stack()[-3]       # 每次写入都提取调用栈
    with open(state_path.replace('STATE.json', '_state_write.log'), 'a') as _lf:
        _lf.write(f'[{_time.strftime("%H:%M:%S")}] WRITE_STATE ...')
```

✅ **正确写法**：
```python
_DIAG_LOG = os.environ.get('ENGINE_DIAG_LOG')  # 环境变量控制开关

def _diag_log(state_path, state):
    if not _DIAG_LOG: return  # 未开启时零开销
    # 诊断逻辑独立函数...

def _write_unlocked(state_path, state):
    _diag_log(state_path, state)  # 一行调用，不污染主逻辑
    # ... 正常写入逻辑
```

### S5-重复模式

❌ **问题代码**：`output()` 函数在 router.py、orchestrator.py、step.py 中各有一份几乎相同的实现

✅ **正确写法**：抽取到公共模块（如 `state_io.py` 或新建 `engine_common.py`），各文件 `from engine_common import output`

### C3-重复函数名

❌ **问题代码**：`filelock.py` 中 `acquire_lock` 函数名出现两次（`fcntl` 版和 `msvcrt` 版）

✅ **正确写法**：跨平台兼容的必要设计，使用 `try/except ImportError` 区分——这是**例外**，不算代码异味

### Y4-职责混合

❌ **问题代码**：`router.py` 的 `main()` 函数兼做参数解析 + 路由查找 + JOIN 检查 + 完成判定（300+ 行）

✅ **正确写法**：拆分为 `_parse_args()` + `_find_candidates()` + `_check_join()` + `_check_complete()` 等单一职责函数
