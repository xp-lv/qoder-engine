# 证伪方法论

> 本文档定义三支红队的证伪策略与判定标准。
> 核心原则：**红队的目标是"推翻"而非"审查"**——只有推翻失败才放行。

---

## 第一支红队：漏洞证伪红队

### 对抗目标

**尽力证明漏洞发现者清单中的每一项是伪阳性。**

### 证伪策略

| 策略 | 描述 | 适用场景 |
|---|---|---|
| **误报排除** | 追踪代码实际执行路径，证明"问题"代码在实际运行中不会被触发 | 死代码分支中的低效实现 |
| **功能必要** | 证明看似低效/不优雅的代码有其功能性原因 | 兼容性补偿、边界条件处理、异常路径保障 |
| **历史约束** | 证明该代码是特定历史背景下的合理决策，当前不能动 | 线程安全补偿、跨平台兼容、渐进式迁移残留 |
| **影响范围验证** | 证明该"问题"的实际影响极小，不值得修改 | 极少执行的路径中的微小低效 |
| **修复风险证明** | 证明修复该问题的风险大于收益 | 修改会触及核心不变量但收益微小 |

### verdict 判定规则

- **confirmed**（证伪失败 → 漏洞为真）：对清单中的每一项漏洞，红队都无法证明其为伪阳性。至少存在一项漏洞是确凿的真实问题。
- **disproved**（证伪成功 → 伪漏洞）：红队成功证明了清单中**所有**漏洞项都是伪阳性。清单中没有任何一条是真实问题。

### 关键约束

- 必须逐项分析漏洞清单中的每一个 finding，不能跳过
- 证伪成功需要给出具体的代码路径证据
- 证伪失败只需对至少一项漏洞确认"找不到反驳理由"

## 第二支红队：方案审查红队

### 对抗目标

**尽力证明修复方案会引入新问题，或方案不优雅/不是机制级修复。**

### 证伪策略

| 策略 | 描述 | 适用场景 |
|---|---|---|
| **连锁影响追踪** | 追踪方案改动会影响哪些调用方/依赖链 | 修改公共函数的行为 |
| **不变量冲突** | 证明方案违反了引擎的已知不变量 | 修改 state_txn 的锁语义 |
| **优雅度挑战** | 证明方案是"打补丁"而非"机制级修复" | 只是在症状层面加 if/else 而非根因修复 |
| **副作用发现** | 证明方案引入了新的副作用 | 新增的环境变量/全局状态/文件写入 |
| **替代方案** | 提出更优雅的替代方案来证明当前方案不够好 | 展示更简洁的实现路径 |

### verdict 判定规则

- **approved**（证伪失败 → 方案安全优雅）：红队无法证明方案有任何问题——不会引入新错误、方案优雅、是机制级修复。
- **rejected**（证伪成功 → 方案有问题）：红队成功证明了方案至少存在以下一个问题：
  - 会引入新错误（连锁影响/不变量冲突/副作用）
  - 不优雅（elegance_score ≤ 4，属于打补丁）
  - 不是机制级修复（只在症状层面处理而非根因）

### 关键约束

- 必须逐个审查方案中的每一个 change item
- rejected 时必须给出具体的反驳理由和建议方向
- elegance_score 评估必须基于实际代码分析，不能给"感觉分"

## 第三支红队：执行审查红队

### 对抗目标

**尽力证明代码执行者的实现引入了错误。**

### 证伪策略

| 策略 | 描述 | 适用场景 |
|---|---|---|
| **语法验证** | 执行 `python3 -c "import ast; ast.parse(...)"` 检查语法 | 基础语法检查 |
| **方案偏差检测** | 对比执行报告中的 diff 与修复方案，找出不一致 | 执行者遗漏了方案中的某项改动 |
| **行为等价验证** | 对比修改前后的 AST 结构，确认行为等价性 | 确认重构保持了行为不变 |
| **新异味扫描** | 对修改后的代码重新执行代码异味扫描 | 重构引入了新的代码异味 |
| **import 链检查** | 确认无断链的 import / 无循环引用 | 重构改动了模块依赖 |
| **__pycache__ 清理验证** | 确认执行者清理了 __pycache__ | 避免缓存干扰 |

### verdict 判定规则

- **clean**（证伪失败 → 执行正确）：红队无法证明执行引入了任何错误——语法正确、与方案一致、行为等价、无新异味。
- **has_defects**（证伪成功 → 执行有错）：红队发现了至少一个执行错误：
  - 语法错误
  - 与方案不一致的实现偏差
  - 行为等价性破坏
  - 新引入的代码异味
  - import 链断裂

### 关键约束

- 必须实际读取修改后的代码文件，不能只看执行报告
- has_defects 时必须标注具体文件、行号和错误描述
- clean 判定必须有"已检查且未发现问题"的正面证据，不能是"没检查到"

## Worked Examples（正反例对照）

### ✅ 正确证伪示例（漏洞证伪红队）

**漏洞清单 finding**：`filelock.py` 中 `acquire_lock` 和 `release_lock` 函数名重复出现两次（C3 检查项）

**正确证伪**：追踪代码发现 filelock.py 使用 `try/except ImportError` 区分 `fcntl`（Unix）和 `msvcrt`（Windows）两个平台实现，同名函数是跨平台兼容的**必要设计**，不是 bug。
- 策略：功能必要
- 证据：`try: import fcntl ... except: import msvcrt`
- 结论：**证伪成功**，此 finding 为伪阳性

### ❌ 错误证伪示例（漏洞证伪红队）

**漏洞清单 finding**：`state_io.py:42-51` 中 `_write_unlocked` 每次写入 STATE.json 都执行 traceback 提取和日志文件追加

**错误证伪**："这个日志看起来是调试用的，可能不需要管"——没有追踪代码路径，没有分析执行频率，仅凭感觉判断。
- 问题：缺乏代码路径证据，仅凭主观判断
- 结论：**证伪失败**，此 finding 是真实问题

### ✅ 正确方案审查示例（方案审查红队）

**方案**：将 state_io.py 的诊断日志提取为 `_diag_log()` 函数，用环境变量 `ENGINE_DIAG_LOG` 控制开关

**正确审查**：追踪发现环境变量未设置时 `_diag_log` 直接 return，等价于注释效果。方案不会改变 STATE.json 写入行为，属于机制级修复。
- 策略：连锁影响追踪 + 优雅度挑战
- 结论：**证伪失败**，方案安全优雅 → approved

### ❌ 错误方案审查示例（方案审查红队）

**方案**：在 `_write_unlocked` 中加 `if os.environ.get('DEBUG'): ...` 条件包裹日志代码

**错误审查**："看起来没问题"——未追踪环境变量名不一致风险，未评估是否有更优雅的替代方案。
- 问题：未挑战优雅度，未提出替代方案
- 结论：这是不合格的审查，应该发现 `DEBUG` 是通用变量名可能被其他模块污染
