# 需求描述

## 目标
构建一个实现型多角色 APP 构建器（app-builder-v2），将上游 pure-arch-design 产出的架构蓝图（Markdown）填充为完整可运行的多角色 APP 包。app-builder-v2 不做架构设计、不做拓扑决策——拓扑决策全部来自上游蓝图，本工具照图施工。

## 上游输入
- **架构蓝图**：`outputs/架构蓝图.md`（由 pure-arch-design 产出，含 7 章节结构：系统总体/层划分/模块清单+RACI/并行兼容性/文档层级/失败模式/对抗记录）

## 核心主线（五项阶段能力）
1. **蓝图解析**（L1）：将 Markdown 蓝图解析为机器可消费的内部结构模型
2. **结构生成**（L2）：依据内部模型生成 APP 包 4 个骨架文件（app.yaml/manifest.json/ROUTER.json/registry.json）
3. **skill/knowledge 填充**（L3）：为每个角色产出 skill.md+schema.json 与领域知识文档
4. **验证**（L4 编译预检）：对完整 APP 包执行 compiler.py --check 编译级结构预检
5. **终审**（L4 语义对抗）：三维红队（忠实度/完整性/一致性）并行对抗校验 + 全局终审裁决

## 四层架构
| 层 | 名称 | 角色 | 层类型 |
|----|------|------|--------|
| L1 | 解析层 | 蓝图解析师 | 封闭层 |
| L2 | 生成层 | 结构生成师 | 封闭层 |
| L3 | 填充层 | Skill填充师 ∥ Knowledge填充师 | 开放层 |
| L4 | 校验层 | 编译预检师 → 忠实度/完整性/一致性红队 → 终审裁决者 | 开放层 |

## 对抗性要求（三维红队）
- **忠实度红队**（M6）：验证生成的 skill 内容是否忠实于蓝图中的角色职责描述
- **完整性红队**（M7）：验证蓝图声明的所有角色/knowledge/产出物是否都已完整生成
- **一致性红队**（M8）：验证 skill 之间引用的文件路径/verdict 名称是否一致

## 并行兼容性约束
- 填充阶段并行度 = 2（M3∥M4，写目录隔离）
- 校验阶段并行度 = 3（M6∥M7∥M8，只读校验）
- 全拓扑最大并行度 = 3 ≤ 3（引擎可靠并行上限）

## 控制流回退机制
- M5 编译预检发现编译缺陷 → 回退 L3 填充层重跑（max_executions: 3）
- M9 终审发现一致性问题 → 回退 L3 填充层修复（max_executions: 3）
- 回退不收敛兜底（rollback_counter > 3）→ M9 标注 consistency_defect 终态退出

## 最终交付物（目标 APP 包）
- `app.yaml` / `manifest.json` / `ROUTER.json` / `registry.json`（结构骨架）
- `roles/{角色}/skill.md` + `schema.json`（每角色执行指令与输出约束）
- `knowledge/{知识名}.md`（领域知识文档）
