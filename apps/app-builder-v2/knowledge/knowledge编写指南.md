# knowledge 编写指南

> 注入目标：Knowledge填充师（M4）
> 用途：指导 M4 为目标 APP 每个角色沉淀领域知识文档

## 目的
本指南定义 knowledge 文档的标准结构、分类体系与 path 对齐规则，确保填充产出的知识文档可被正确注入角色上下文且 Gate 校验通过。

## 核心准则

### 1. knowledge 文档四类分类
| 类型 | 用途 | 示例 |
|------|------|------|
| 方法论文档 | 指导"怎么做"的推理链与判据 | 层分解方法论、模块划分原则 |
| 格式规范文档 | 定义"长什么样"的标准结构 | 文档格式规范、编号规则 |
| 编写指南文档 | 约束"写多细"的颗粒度 | 实现规格编写指南、追溯链方法 |
| 领域知识文档 | 特定业务域的约束与术语 | 业务规则、术语表、接口规范 |

### 2. 文档五段结构（强制）
每份 knowledge 文档必须包含：
- `# 标题`
- `> 注入目标：xxx` + `> 用途：xxx`（frontmatter）
- `## 目的`：本文档解决什么问题
- `## 核心准则`：3-7 条编号准则（含 Why 推理链）
- `## 判别清单`：可操作的 checkbox 核对项
- `## 反模式`：常见错误与后果（3-5 条）

### 3. path 对齐规则（关键）
- knowledge 文档的 path 必须与 app.yaml knowledge 段完全对齐
- 不一致会导致：① init.py 拷贝失败 ② Gate 校验失败 ③ 角色执行时找不到注入文件
- M4 产出前必须逐一核对 app.yaml knowledge 段的每个 path

### 4. inject_to 对齐规则
- inject_to 中的角色名必须在 app.yaml roles 中存在
- 一个 knowledge 文档可注入多个角色（inject_to: [角色A, 角色B]）
- 一个角色可注入多份 knowledge（在 app.yaml knowledge 段多次声明）

### 5. 内容深度要求（方法论升级）
方法论文档不应仅停留在"形式化约束"（What/How），应升级为"方法论指南"：
- **Why 推理链**：为什么这条准则成立（经典理论出处，如 Dijkstra / Miller's Law）
- **When not**：什么情况下不适用（判别边界）
- **省略后果**：不遵循会导致什么具体问题
- **认知科学基础**：心理学机制（如防御性话话 / 锚定效应）

## 判别清单

M4 填充时逐项核对：
- [ ] app.yaml knowledge 段声明的每个 path 都有对应的 .md 文件？
- [ ] 每份文档含目的/核心准则/判别清单/反模式四段？
- [ ] inject_to 角色名都在 app.yaml roles 中？
- [ ] 方法论文档含 Why 推理链（不只是 What）？
- [ ] 每份文档 ≥ 20 行实质性内容？

## 反模式
- ❌ knowledge path 与 app.yaml 不对齐（Gate 校验失败）
- ❌ 方法论文档只有"必须这样做"没有"为什么"（缺乏思想纵深）
- ❌ 一份 knowledge 塞进所有角色的所有知识（应按 inject_to 精准注入）
- ❌ 文档内容空洞（只有标题没有准则/清单/反模式）
