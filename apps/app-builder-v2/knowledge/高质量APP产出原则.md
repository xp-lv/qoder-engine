# 高质量 APP 产出原则

> 注入目标：Skill填充师（M3）、Knowledge填充师（M4）、结构生成师（M2）、三红队（M6/M7/M8）、终审裁决者（M9）
> 用途：app-builder-v2 产出的目标 APP 必须满足的质量底线——这不是格式规范，而是思想标准

## 目的

本原则是 app-builder-v2 的"质量宪法"。它来源于《工作流编排原则》文档族的 7 条核心原则，回答一个根本问题：**什么样的 APP 是"好的 APP"？**

答案不是"编译通过的 APP"，而是**"让每个角色都能专注于自己核心能力的 APP"**。

> 编排是为"专注"服务的。专注包括产出、检查、决策——它们都是角色的核心能力。真正的敌人是引擎约束和修复说明对 skill 的污染。
> —— 《00-编排哲学总纲》

## 核心准则

### 准则 1：重点意识——核心能力占比 ≥ 60%

**Why（为什么这条准则成立）**：
LLM 的上下文窗口是有限资源（认知科学的"工作记忆"约束，Miller's Law：7±2 个信息块）。如果 skill 中 60% 以上是引擎修复说明和跨角色机制描述，角色的"工作记忆"被编排税占满，核心能力被挤出——角色不知道自己该专注什么。

**Who（谁负责）**：
- **M3 Skill填充师**：产出 skill 时确保核心能力描述 ≥ 60%，引擎约束/修复说明 = 0%（理想）
- **M6 忠实度红队**：校验产出的 skill 核心能力占比是否达标

**How（怎么判别）**：
对 skill.md 逐行分类：
```
[能力] 产出方法论（怎么产出、产出什么形态）         ← 核心能力
[能力] 检查方法论（检查什么、按什么原则定级）       ← 核心能力
[能力] 决策方法论（verdict 选项、每个对应什么）     ← 核心能力
[能力] 文件使用方法论（从注入文件提取什么、怎么用） ← 核心能力（教 role 干活）
[税]   引擎修复说明（C1/M2 修复必读）              ← 编排税
[税]   跨角色机制描述（其他角色怎么配合我）         ← 编排税
[税]   引擎约束解释（SDK §X.Y 的限制）              ← 编排税
[税]   元流程描述（整个工作流拓扑怎么走）           ← 编排税
```
**核心能力占比 = 核心能力行数 / 总行数 ≥ 60%**。低于 60% = 编排税污染。

**反模式**：pure-arch-design 的终审裁决者 skill 277 行，核心能力仅 40 行（14%），编排税占 86%。

### 准则 2：编排税意识——引擎的事归引擎，角色的事归角色

**Why**：
引擎修复说明（如"C1 修复必读"）和引擎约束解释（如"SDK §3.4 规定 fail 边…"）是引擎演进的产物。每次引擎升级，所有包含这些内容的 skill 都要跟着改——这是维护噩梦。更严重的是，这些内容与角色核心能力无关，挤占了本已有限的 skill 篇幅。

**Who**：
- **M3 Skill填充师**：产出 skill 时绝不写引擎修复说明、引擎约束解释、跨角色机制描述、元流程描述
- **M8 一致性红队**：校验 skill 是否被编排税污染

**How**：
skill 中出现以下内容即为编排税，必须消除：
- ❌ "C1/M2/M3 修复必读" → 移到引擎文档
- ❌ "SDK §X.Y 规定…" → 由 dispatch.schema_constraints 注入
- ❌ "其他角色会怎么配合我" → 由 edges/carries 在 app.yaml 声明
- ❌ "本工作流有 N 个角色通过 N 轮对抗…" → 直接删除

**但注意**：以下内容**不是**编排税，是核心能力，必须保留：
- ✅ 产出方法论（怎么产出、产出什么形态）
- ✅ verdict 判定规则（这是决策方法论）
- ✅ 文件使用方法论（从注入文件提取什么、怎么用——这是教 role 干活）

### 准则 3：独立视角价值——检查角色因"无滤镜"而有独立价值

**Why（认知科学基础）**：
产出者天然有 3 种认知偏差：
1. **确认偏差**（Confirmation Bias）：倾向于寻找支持自己结论的证据——"我写的应该没问题"
2. **锚定效应**（Anchoring）：被自己已有的设计锚定——"用了 A 方案就不会想到 B 方案可能更好"
3. **知识诅咒**（Curse of Knowledge）：自己知道的东西难以想象"不知道会怎样"

检查角色的核心能力 = **无滤镜的独立视角**。这种能力只能通过独立的上下文（subagent）实现，**产出者的自检无法替代**——这是检查角色存在的根本理由，不是冗余。

**Who**：
- **M2 结构生成师**：生成 app.yaml 时，确保检查角色与产出角色信息隔离（inputs 不含产出者的思考过程）
- **M6/M7/M8 三红队**：自身保持信息隔离和利益对立（"职责是证明产出不行，不是帮忙改进"）

**How**：
- 信息隔离：检查角色 inputs = [最终产出物]，**绝不包含**产出者的设计笔记/推理过程
- 利益对立：检查角色 skill 必须声明"你的职责是证明它有问题"
- 原则外化：判定依据来自外部资产（检查清单/严重度准则），不是角色自己推理

### 准则 4：资产优先——能用清单解决的不要用编排

**Why**：
资产（检查清单/模板/模式库）的成本曲线是"一次性高，之后趋近于 0"；编排（角色/verdict/edge/回退）的成本曲线是"每次执行都付，且随复杂度线性增长"。当二者都能达成同一目的时，优先用资产。

**Who**：
- **M4 Knowledge填充师**：优先把检查项固化为清单资产、把产出形态固化为模板资产
- **M7 完整性红队**：校验目标 APP 的资产（清单/模板）是否完整

**How**：
设计任何机制时，按此顺序决策：
1. 能用**检查清单**解决？→ 沉淀为清单资产
2. 能用**模板**解决？→ 沉淀为模板资产
3. 能用**模式库**解决？→ 从模式库选取
4. 必须动态决策？→ 用编排解决，但要最小化

**反模式**：用 3 轮红队（编排解）做"REQ-ID × 模块矩阵"（检查清单）能做的事——编排解成本 = 数十次 LLM 调用，清单解成本 = 对照打勾 1 次。

### 准则 5：职责聚焦——一个角色一个核心能力

**Why**：
一个角色承担 ≥ 3 类不相关职责时，其 skill 必然臃肿（上帝角色反模式），核心能力被稀释。编排的目的是"让每个角色专注"，如果一个角色自己都不专注，编排就失去了意义。

**Who**：
- **M2 结构生成师**：生成 app.yaml 时，确保每个角色只有一个核心能力声明
- **M6 忠实度红队**：校验目标 APP 中是否有上帝角色

**How**：
对每个角色问：**能用一句话写出它的核心能力吗？**（无论它是产出、检查还是决策）
- 如果一句话写不完 → 承担了多类职责 → 拆分
- 如果多个角色的核心能力声明高度相似 → 角色镜像 → 合并或差异化

### 准则 6：Build-Run 分层——搭管线 ≠ 跑管线

**Why**：
很多工作包含两个性质完全不同的阶段：
- **Build**（搭建期）：低频、产可复用资产、工匠心智（精雕细琢）
- **Run**（运行期）：高频、产一次性结果、探索者心智（快速试错）

把两者混在一个工作流，会导致 Build 内容占 70% 的 skill 篇幅，Run 业务被挤到 30%——违背专注度。

**Who**：
- **M2 结构生成师**：生成 app.yaml 时，评估目标 APP 是否需要 Build-Run 分层
- **M1 蓝图解析师**：解析蓝图时，识别 Build 产出和 Run 产出

**How**：
问：**这件事的产出，下次还会用到吗？**
- "会反复用" → Build 产出 → 考虑独立为资产或独立 app
- "用完就完了" → Run 产出 → 专注核心业务

### 准则 7：耦合点显性化——verdict 是必要的耦合点

**Why**：
verdict 是 skill 与 dispatch 的必要耦合点，无法避免，也不应该避免。每个 verdict 对应角色核心能力的一个决策分支——这是决策方法论的核心，不是编排税。但如果 verdict 是为绕过引擎限制而设计的（如用不同 verdict 区分"同一角色不同上下文"），那就是编排税。

**Who**：
- **M3 Skill填充师**：产出 skill 时，让 verdict 声明成为决策方法论的一部分
- **M8 一致性红队**：校验 verdict 是否为必要耦合点（非绕过引擎限制）

**How**：
判别每个 verdict：
- ✅ **必要的 verdict**：对应角色核心能力的一个真实决策分支（如 passed/failed/needs_revision）
- ❌ **编排税 verdict**：为绕过引擎限制而设计（如因引擎不支持"同一角色不同上下文"而用不同 verdict 区分）

## 编排三问（每个角色产出前必须回答）

任何编排设计开始前，必须对照回答：
1. **这次编排服务的是哪个目的？**（拆解/隔离/质量/规范/专注/无意识路由——说不清楚就是"为编排而编排"）
2. **这个目的能不能不用编排达成？**（能用清单/模板/模式库解决吗？）
3. **这是角色核心能力，还是引擎约束？**（前者保留，后者消除）

## 反模式速查（产出 APP 时必须避免）

| # | 反模式 | 症状 | 违背原则 |
|---|--------|------|---------|
| 1 | 上帝角色 | 一个角色承担 4+ 类职责 | 职责聚焦 |
| 2 | 编排万能论 | 什么问题都加角色/加 edge | 资产优先 |
| 3 | 修复必读污染 | skill 里大量"XX 修复必读" | 编排税意识 |
| 4 | verdict 爆炸 | 一个角色 5+ verdict，部分为绕过引擎 | 编排税意识 |
| 5 | 角色镜像 | 多个检查角色 skill 90% 相同 | 独立视角价值 |
| 6 | Build-Run 混淆 | 搭管线和跑管线塞在一起 | Build-Run 分层 |
| 7 | 跨角色机制描述 | skill 描述"别人怎么配合我" | 编排税意识 |
| 8 | 完整性优先 | 为覆盖边界牺牲核心能力占比 | 重点意识 |
| 9 | 元流程自解释 | skill 描述工作流全局拓扑 | 编排税意识 |
| 10 | 路径散落 | skill 里到处写文件地址 | 重点意识 |

## 判别清单

app-builder-v2 的每个角色产出前，逐项核对（从 7 条原则中选取与自身相关的）：
- [ ] 产出的 skill 核心能力占比 ≥ 60%？（准则 1）
- [ ] 产出中不包含引擎修复说明/引擎约束解释？（准则 2）
- [ ] 检查角色与产出角色信息隔离？（准则 3）
- [ ] 能用清单/模板解决的不用编排？（准则 4）
- [ ] 每个角色一个核心能力（一句话能写完）？（准则 5）
- [ ] Build 产出和 Run 产出是否需要分层？（准则 6）
- [ ] verdict 是核心决策而非绕过引擎限制？（准则 7）

## 一句话总结

> **好的 APP 不是"编译通过的 APP"，而是"让每个角色都能专注于自己核心能力的 APP"。编排税是专注的头号敌人，资产是质量提升的主力，独立视角是检查角色存在的根本理由。**
