# 蓝图解析规范

> 注入目标：蓝图解析师（M1）
> 用途：指导 M1 如何将上游架构蓝图（Markdown）解析为内部结构模型（JSON）

## 目的
本规范定义 app-builder-v2 的 L1 解析层如何从 pure-arch-design 产出的架构蓝图中提取结构化信息，确保下游结构生成师能机械消费内部模型，无需二次理解 Markdown 语义。

## 核心准则

### 1. 蓝图七章节结构（强制）
上游蓝图必须包含以下 7 个一级章节（`## ` 标题），缺一不可：
1. `## 1. 系统总体` — 系统目标、核心能力、上下文图
2. `## 2. 层划分` — 层架构表、层间依赖、层类型（封闭/开放）
3. `## 3. 模块清单 + RACI` — 角色清单、RACI 矩阵、产出物路径契约、各模块详解
4. `## 4. 并行兼容性` — 引擎可行性分析、资源访问约束
5. `## 5. 文档层级` — 文档层级图、文档生长规则
6. `## 6. 目标架构的失败模式与应对` — 单点失效、依赖不可用、过载
7. `## 7. 对抗记录` — 红队问题清单与架构师响应

### 2. REQ-ID 落位三要素
蓝图中每个 REQ-ID 必须满足落位最小形态三要素：
- REQ-ID 显式引用（`REQ-XXX` 字样）
- 每条 AC（acceptance_criteria）逐条响应（接受/拒绝 + 理由）
- 落位结论句（"REQ-XXX 已由本节 §X.X 完整承担"）

### 3. 内部模型四段结构
解析产出的内部蓝图模型（JSON）必须包含四段：
- `entry_check`：入口门禁结果（passed / chapter_count / req_ids）
- `roles`：角色数组（id / name / capability / layer / layer_type / responsibility / knowledge_refs）
- `topology`：拓扑信息（edges / forks / joins）
- `deliverables`：产出物数组（name / path / format / owner）

## 判别清单

解析时逐项核对：
- [ ] 蓝图 7 章节标题全部 grep 命中？
- [ ] 每个 REQ-ID 有落位结论句？
- [ ] roles 数组中每个角色的 layer 字段值在 {L1, L2, L3, L4} 中？
- [ ] topology.forks 中每个 FORK 的 targets 数 ≤ 3（引擎并行上限）？
- [ ] deliverables 中每个产出物有明确的 owner（指向某角色 id）？

## 反模式
- ❌ 跳过入口门禁直接解析（会导致无效产出传递给下游）
- ❌ 在内部模型中添加蓝图未声明的角色/拓扑（M1 是解析者不是设计者）
- ❌ 解析时做语义判断（如"这个角色设计不合理"）——语义判断归 M6/M7/M8 红队
