# 约束分层规范

> 本规范明确各类约束应该放在哪一层文件中，避免"硬约束放错位置被重新编译覆盖"或"软约束硬编码到派生文件"的问题。
>
> 源自评审中发现的真实问题：schema.json（派生文件）被手工加 minLength/findings items 等软约束，compiler 重新编译后被覆盖。
>

## 一、核心原则

**约束放对位置 = 单一权威源 + 派生文件不被破坏**

| 文件类型 | 性质 | 约束类型 |
|---------|------|---------|
| **app.yaml** | 唯一权威源（声明式）| 拓扑/角色/knowledge 注册/edges |
| **schema.json** | **半派生文件**（compiler 生成基础，保留手写 contract）| _required_files（含可选手写 contract）|
| **registry.json** | **派生文件**（compiler 生成）| verdicts + inputs + outputs 等编译产物 |
| **ROUTER.json** | **派生文件**（compiler 生成）| transitions + carries（边级显式声明编译）|
| **skill.md** | **内容文件**（手写，compiler 不覆盖）| 执行步骤 + 自检项 + SDK陷阱警告 |
| **principles.md** | **内容文件**（手写）| 设计原则 + 校验清单 |
| **knowledge/*.md** | **内容文件**（手写）| 业务规则 + 流程治理 + SDK陷阱 |

## 二、约束分层表（什么放哪里）

### 2.1 派生文件层（compiler 生成，禁止手改）

| 文件 | 允许内容 | 禁止内容 |
|------|---------|---------|
| **schema.json** | • `_required_files`（从 outputs 提取）<br>• `contract`（手写深度校验，compiler 保留不覆盖）| ❌ `verdict.enum`（v9.2 删除，权威源迁移到 ROUTER.json transitions）<br>❌ `minLength`（软约束）<br>❌ `findings.items`（软约束）<br>❌ `additionalProperties`（软约束）|
| **registry.json** | • `verdicts` 数组（compiler 从 edges 提取）<br>• `outputs` / `inputs` / `gate_rules` / `input_groups` | ❌ 任何手写字段 |
| **ROUTER.json** | • `transitions`（含 fail 边自动注入）<br>• `carries`（从 app.yaml 边级显式声明编译）<br>• `verdict_context` | ❌ 手工加边<br>❌ 手工改 fail target |

### 2.2 内容文件层（手写，compiler 不覆盖）

| 文件 | 允许内容 | 典型约束 |
|------|---------|---------|
| **skill.md** | 执行步骤 + 自检项 + verdict 表 + SDK陷阱警告 | "summary ≥ 50 字符" / "findings 必须含 check_id" |
| **principles.md** | 设计原则 + 校验清单表格 | "每条 problem 必须引用严重度准则编号" / "default severity=high" |
| **knowledge/*.md** | 业务规则 + 量化约束 + 正反例 | "acceptance_criteria 禁用模糊词" / "REQ-ID 正则 ^REQ-\d{3,4}$" |

### 2.3 app.yaml（唯一权威源）

| 允许内容 | 说明 |
|---------|------|
| `app_name` / `knowledge` / `roles` / `edges` | 声明式拓扑 |
| 角色的 `confirm` / `inputs` / `outputs` / `fail_max_executions` | 角色定义 |
| `restrict_verdict` / `max_executions` / `carries` | 边级元数据 |

## 三、约束三层模型（v9.2）

### 硬约束（Gate 强制校验）

| 约束 | 校验层 | 权威源 | 失败行为 |
|------|--------|--------|----------|
| **verdict 合法性** | Gate Layer 0（envelope）| ROUTER.json transitions keys | ENVELOPE_FAIL → BLOCKING |
| **产出物文件存在** | Gate Layer 1（file）| schema.json `_required_files` | FAIL → 走 fail 边 |
| **产出物非空** | Gate Layer 1（file）| — | FAIL → 走 fail 边 |

> **v9.2 变更**：verdict 校验权威源从 schema.json `result.verdict.enum` 迁移到 ROUTER.json transitions keys（Gate Layer 0 直接读取）。schema.json 不再包含 verdict enum。

### 半硬约束（Gate Layer 1 contract 深度校验）

contract 是 schema.json 中**唯一允许手写的内容**，compiler 重新生成时保留已有 contract 不覆盖。

| contract 字段 | 校验语义 |
|---------------|----------|
| `min_lines` | 文件行数 ≥ N |
| `required_headings` | 标题/关键字必须在文档中出现 |
| `req_coverage` | 每个 REQ-ID 必须在文档中出现 |
| `forbidden_patterns` | 文档不得包含指定字符串 |

### 软约束（运行期 LLM 自检，放内容文件）

- **summary minLength**：skill 自检项
- **findings items schema**：principles 校验清单
- **量化指标**（REQ-ID 正则 / 模糊词禁用）：knowledge 业务规则
- **default severity**：principles 校验清单表格

**为什么软约束不放在 schema.json？**
1. contract 只支持简单文本匹配（行数/标题/REQ-ID），不适合复杂业务规则
2. 软约束需要业务上下文解释，放内容文件让 LLM 理解和执行更可靠
3. findings 结构校验用 principles 校验清单表格比 JSON Schema 更直观

## 四、违规案例与正确做法

### 案例 1：手工加 minLength 到 schema.json

```json
// ❌ 错误（派生文件，重新编译被覆盖）
"summary": {
  "type": "string",
  "minLength": 50
}
```

```markdown
<!-- ✅ 正确（内容文件，compiler 不覆盖）-->
## 自检项
- [ ] result.summary ≥ 50 字符？
```

### 案例 2：手工加 findings.items 到 schema.json

```json
// ❌ 错误
"findings": {
  "type": "array",
  "items": {
    "type": "object",
    "required": ["check_id", "severity", "description"]
  }
}
```

```markdown
<!-- ✅ 正确（principles.md §二 校验清单）-->
| # | check_id | 检查内容 | 失败判据 | 默认 severity |
|---|----------|---------|---------|--------------|
| 1 | 必填字段 | ... | ... | high |

### 校验执行规则
- 失败项填入 findings（含 check_id / severity / description / evidence / suggested_fix）
```

### 案例 3：手工加 verdict enum 到 schema.json

```json
// ❌ 错误（v9.2：schema.json 不再包含 verdict enum）
"result": {
  "properties": {
    "verdict": { "enum": ["confirmed", "challenged"] }
  }
}
```

verdict 合法性由 Gate Layer 0 从 ROUTER.json transitions keys 读取，不需要在 schema.json 中声明。

## 五、检查流程（每次修改约束时必做）

```
1. 这是硬约束还是软约束？
   ├─ 硬约束（verdict enum / 文件存在性）→ 派生文件（compiler 自动）
   └─ 软约束（字段内容 / 量化指标）→ 内容文件（手写）

2. 这份文件是派生文件还是内容文件？
   ├─ 派生文件（schema/registry/ROUTER）→ 禁止手改，改 app.yaml + 重新编译
   └─ 内容文件（skill/principles/knowledge）→ 可手写

3. 这个约束是否已经在其他地方声明过？
   ├─ 是 → 改为引用（单一权威源）
   └─ 否 → 在最合适的层声明
```
