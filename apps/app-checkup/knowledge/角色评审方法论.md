# 角色评审方法论（6 维 × 8 步）

> 本文档是评审 app 角色设计（skill / principles / schema / knowledge）的可复用方法论。
> 源自 7 轮实战评审（pure-arch-design-app-builder 全部 8 个角色），提炼为显式可教学框架。
>
> **适用场景**：
> - 评审新建 app 的角色设计
> - 评审现有 app 的迭代优化
> - 审阅角色（结构审阅者 / 合规审阅者等）的执行依据
> - 跨 app 质量对标
>
> **资产路径**：/Users/xiaopenglv/ai-agents/多角色配合系统/z-workspace/工作流编排原则/审计原则/角色评审方法论.md

## 一、方法论来源

本方法论源自两个底层共识：

1. **机制修复双重判定**：消除多源冲突 + 错误不掩盖
2. **资产优先 + 重点占比 ≥60%**：业务核心与编排税分离

从这两个共识出发，经过两轮实战评审验证，提炼为 **5 维 × 8 步** 框架。

## 二、5 维度评审框架（按优先级降序）

```
B 一致性  ━━━━━━━━━━━━━━━━━━━  最高优先级（所有真实 bug 都在这里）
F 契约    ━━━━━━━━━━━━━━━━━   dispatch↔skill 接口一致性
C 完备性  ━━━━━━━━━━━━━━━━━━
D 质量    ━━━━━━━━━━━━━━━━
E 机制    ━━━━━━━━━━━━━━━
A 职责    ━━━━━━━━━━━━━━        最低优先级（基本不会出错）
```

> **优先级排序原则**：
> - **B 最高**：真实 bug 全部来自多源冲突——同一个信息在多个文件中出现，必然漂移
> - **A 最低**：职责边界通常在设计阶段已确定，执行评审时基本不会出错

### A 职责维度（最低优先级）

| 编号 | 检查项 | ✅ 正例 | ❌ 反例 |
|------|--------|---------|---------|
| **A1** | 角色职责单一（符合"角色即能力"原则） | 需求分析师只做需求分析 | 一个角色同时做"分析 + 校验" |
| **A2** | 不越界（不承担其他角色职责） | 架构师只做架构设计 | 架构师做了需求分析的工作 |
| **A3** | confirm 属性符合 SDK_SPEC（v9.2：type 字段已废弃）| manual 角色 confirm: manual | 声明了已废弃的 type: producer |

### B 一致性维度（最高优先级——bug 富集区）

| 编号 | 检查项 | 验证手段 |
|------|--------|---------|
| **B1** | **verdict 三方一致性**（v9.2：schema enum 已删除）| registry verdicts ↔ ROUTER transitions ↔ skill verdict 表 |
| **B2** | **产出物三方一致性** | schema `_required_files` ↔ app.yaml `outputs` ↔ registry `outputs` |
| **B3** | **跨文件路径一致性** | grep 所有引用路径（`.md` vs `.json` 后缀、绝对 vs 相对路径） |
| **B4** | **校验项一致性** | skill 自检项 ↔ principles 校验清单段 ↔ schema 必填字段 |

### C 完备性维度

| 编号 | 检查项 |
|------|--------|
| **C1** | **skill 七段结构**（v9.2：五段→七段，详见《skill 编写规范》）：角色定位 / 入口判定 / 执行步骤 / 产出物格式 / 输入消费指南 / verdict 语义表 / 自检项 |
| **C2** | **principles 双段结构**（有校验角色的被校验角色）：设计原则段（产出者用）/ 校验清单段（校验角色用）—— 段间边界显式 |
| **C3** | **schema.json 产物校验**（v9.2：verdict enum 已删除，权威源迁移到 ROUTER.json transitions）：_required_files 存在 + 可选 contract 深度校验 |
| **C4** | **知识引用显式化**：skill 必须列出引用的 knowledge 文档清单（不能隐式依赖） |

### D 质量维度

| 编号 | 检查项 | ✅ 正例 | ❌ 反例 |
|------|--------|---------|---------|
| **D1** | **重点占比 ≥60%** | skill.md 70% 内容是业务执行步骤 | skill.md 60% 内容是 SDK 陷阱和 Changelog |
| **D2** | **单一权威源** | skill.md 引用《校验清单》knowledge | 同一校验规则在 skill.md 和 knowledge 中各写一遍 |
| **D3** | **量化指标** | "summary ≥ 50 字符" / "REQ-ID 匹配 ^REQ-\d{3,4}$" | "写好一点" / "确保质量" |
| **D4** | **正反例** | knowledge 中有 ✅ 和 ❌ 示例 | 纯规则描述无示例 |

### E 机制维度（隐藏 bug 探测）

| 编号 | 检查项 | 典型症状 |
|------|--------|---------|
| **E1** | **fail 边 target 正确性** | 自指（死循环） / 指向不可修正的角色 / 缺失 |
| **E2** | **校验角色显式定义合理性**（v8.0：原 producer 自动展开已废弃）| 校验角色是否在 app.yaml 中显式定义 + edges 是否连接 |
| **E3** | **carries 声明与 skill 消费匹配** | 边写了 carries 但 skill 未说明如何使用 / skill 假设收到上游物料但边未声明 carries |
| **E4** | **principles 双用途边界** | 设计原则段与校验清单段的可见性是否对目标角色正确 |

### F 契约维度（dispatch ↔ skill 接口一致性）

| 编号 | 检查项 | 验证手段 |
|------|--------|---------|
| **F1** | **路径权威源不冲突** | skill 不硬编码产出物/输入文件绝对路径（以 dispatch 为准） |
| **F2** | **verdict 表为全集** | skill verdict 表写全集 + 语义，不写动态过滤后的子集 |
| **F3** | **输入物料声明一致** | skill 描述的输入来源 ↔ app.yaml 边 carries 声明（不假设隐式注入） |
| **F4** | **principles 消费方式正确** | registry 声明了 principles → skill 不自行 Read；未声明 → skill 不假设内联 |

> **F 维度定位**：B 维度查文件之间的静态一致性，F 维度查运行时 dispatch→skill 的动态契约。详见《dispatch-skill 分工规范》

## 三、8 步评审流程

```
┌─────────────────────────────────────────────────────┐
│ Step 1. 通读 app.yaml                                │
│   → 理解整体拓扑 + 角色清单 + edges                  │
├─────────────────────────────────────────────────────┤
│ Step 2. 读 ROUTER.json + registry.json               │
│   → 理解单角色的注册上下文                           │
│   → 关注：verdicts 数组 / transitions keys /         │
│           input_groups / principles 引用路径         │
├─────────────────────────────────────────────────────┤
│ Step 3. 单角色 4 文件齐读（并行）                    │
│   → skill.md / principles.md / schema.json /         │
│     knowledge/*.md                                   │
├─────────────────────────────────────────────────────┤
│ Step 4. A 职责评估                                   │
│   → 定位是否合理（通常此步快速通过）                 │
├─────────────────────────────────────────────────────┤
│ Step 5. C+D 单文件评审                               │
│   → 每个文件内部质量（C 完备性 + D 质量）            │
│   → 输出问题表（编号 / 问题 / 严重度）               │
├─────────────────────────────────────────────────────┤
│ Step 6. B 跨文件扫描（bug 富集区）                   │
│   → grep 批量核验路径/verdict/产出物一致性           │
│   → 典型命令：grep "需求澄清请求" 全目录             │
├─────────────────────────────────────────────────────┤
│ Step 7. E 隐藏 bug 探测                              │
│   → fail 边 target / verdict enum 单值 /             │
│     校验角色显式定义合理性（v8.0：原 producer 自动展开已废弃）/ SDK 陷阱（T1 已失效）│
│   → 用 JSON 校验脚本验证三方一致性                   │
├─────────────────────────────────────────────────────┤
│ Step 8. 评分 + 修复建议                              │
│   → 5 星制评分                                       │
│   → P0/P1/P2 优先级 + 工作量估计                    │
│   → 讨论点（开放问题给用户决策）                     │
└─────────────────────────────────────────────────────┘
```

## 五、可操作 Checklist（精简版）

评审单角色时，按顺序勾选：

### A 职责（快速通过）
- [ ] A1 角色职责单一
- [ ] A2 不越界
- [ ] A3 confirm 属性符合 SDK_SPEC（v9.2：type 字段已废弃）

### B 一致性（bug 富集区，重点扫描）
- [ ] B1 verdict 三方一致（registry/ROUTER/skill）（v9.2：schema 不再含 verdict enum）
- [ ] B2 产出物三方一致（schema/app.yaml/registry）
- [ ] B3 跨文件路径一致（grep 所有路径字符串）
- [ ] B4 skill 自检项 ↔ principles 校验清单对齐

### C 完备性
- [ ] C1 skill 七段结构齐全（角色定位/入口判定/执行步骤/产出物格式/输入消费/verdict表/自检项）
- [ ] C2 principles 设计原则/校验清单段边界显式
- [ ] C3 schema _required_files + contract 校验（v9.2：不再含 verdict enum）
- [ ] C4 skill 显式引用 knowledge 文档清单

### D 质量
- [ ] D1 业务核心占比 ≥60%（识别编排税：SDK 修补说明、Changelog）
- [ ] D2 同一信息单一权威源（识别冗余设计）
- [ ] D3 约束可证伪（正则/字数/条数）
- [ ] D4 业务指南有正反例

### E 机制（隐藏 bug 探测）
- [ ] E1 fail 边 target 不自指、不指向不可修正角色
- [ ] E2 校验 step 有真正"能力"（非 14 行空泛）
- [ ] E3 carries 声明与 skill 消费匹配
- [ ] E4 principles 双用途边界清晰

### F 契约（dispatch ↔ skill 接口）
- [ ] F1 skill 不硬编码路径（以 dispatch 权威路径为准）
- [ ] F2 skill verdict 表写全集 + 语义（不写动态子集）
- [ ] F3 skill 输入说明与 carries 声明匹配
- [ ] F4 principles 消费方式正确（不重复 Read）

## 六、评分标准

| 星级 | 标准 |
|------|------|
| ⭐⭐⭐⭐⭐ (5/5) | 无任何问题，可直接投产 |
| ⭐⭐⭐⭐ (4/5) | 仅 minor 问题，不阻断 |
| ⭐⭐⭐ (3/5) | 有 medium 问题，建议修复 |
| ⭐⭐ (2/5) | 有 high 问题，必须修复 |
| ⭐ (1/5) | 有 critical 问题，不可用 |

## 七、与用户偏好的映射

本方法论的每个维度都对应一条已固化的用户偏好：

| 维度 | 映射偏好 |
|------|---------|
| **B 一致性** | 机制修复双重判定标准（消除多源冲突） |
| **C 完备性** | （无直接对应，属于工程实践共识） |
| **D 质量** | 资产优先 / 重点占比 ≥60% / 冗余设计术语偏好 |
| **E 机制** | carries 声明匹配 / fail 边系统保留 |
| **A 职责** | 正交维度建模偏好 / 角色即能力 |
| **F 契约** | 编译器全权负责确定性物料编排 / dispatch 提供 What/Skill 提供 How |

## 八、实战验证记录

### 已评审案例

| 案例 | 评分 | 关键发现 | 修复后评分 |
|------|------|---------|----------|
| pure-arch-design 需求分析师 | 4/5 | K1 格式漂移 / K2 SDK 污染 / S1 缺警告 | 5/5 |
| pure-arch-design 需求分析师_校验_ | 2/5 | V1 verdict 三方不一致 / fail 边自指 bug | 5/5 |
| pure-arch-design 架构设计师 | 3/5→4/5 | S1 verdict 表语义错误 / P1 principles 未同步 | 5/5 |
| pure-arch-design 结构红队 | 4/5 | S1 verdict 表含 fail / K1.2 与方法论冗余 | 5/5 |
| pure-arch-design 边界红队 / 极限红队 | 4/5 | 同结构红队（模板套用）| 5/5 |
| pure-arch-design 终审裁决者 | 4/5 | C4 _required_files 死循环 / K1+K2+K3 冗余 | 5/5 |

### 高频 bug 模式（实战统计）

| 排名 | 维度 | bug 模式 | 出现频率 |
|------|------|---------|--------|
| 1 | B1 | verdict 多源不一致（schema/registry/ROUTER/skill） | 3/5 |
| 2 | E1 | fail 边 target 自指或指向不可修正角色 | 2/5 |
| 3 | E2 | 校验 step / 多路径角色的产出物机制缺陷 | 2/5 |
| 4 | D1 | SDK 修补说明污染业务文档（编排税超标） | 4/5 |
| 5 | D2 | 同一信息多源重复（跨文档冗余设计） | 5/5 |
| 6 | C2 | principles 未按 §一/§二 双段结构组织 | 2/5 |

> **结论**：**D2（冗余设计）+ D1（编排税）+ B1（verdict 一致性）是评审时最值得投入精力的三个维度**——历史数据显示 80% 的真实 bug 都在这三个维度。

## 十、使用建议

### 评审者角色注入
建议把本文档作为以下角色的知识注入：
- **结构审阅者**：关注 A 职责 + B 一致性
- **合规审阅者**：关注 C 完备性 + D 质量
- **架构红队**：关注 E 机制（隐藏 bug）

### 自审场景
角色设计者在产出后，按 §五 Checklist 自审一轮，可消除 80% 的 B/E 维度 bug。

### 跨 app 对标
不同 app 的相同角色（如所有 app 的"需求接收者"）可对照本方法论做横向质量对比。
