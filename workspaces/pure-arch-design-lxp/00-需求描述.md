## 1. 目标（要做什么）

对 **lxp-eng-planning 应用** 进行架构层面的重新设计。lxp-eng-planning 是一个"工程规划+精力管理"领域的多角色协作 APP，当前为 18 角色前后端分离架构。

本次重新设计的**核心目标：迭代能力重构**。

### 1.1 当前架构的迭代缺陷

现有迭代机制存在以下结构性问题：

1. **迭代入口过于粗糙**：当前由"迭代守门人"用 `fresh_start / incremental / done` 三选一驱动迭代，粒度过粗——无法支持"仅前端迭代"、"仅后端迭代"、"仅文档层迭代"等部分增量场景。
2. **迭代反馈回路过长**：迭代反馈需经过"全栈联调验证者（manual）"统一收口，无论改动范围大小都要走完整链路，无法快速试错。
3. **增量基线不明确**：`incremental` 分支跳到"全栈联调验证者"，但增量基线（上一轮产出物如何作为输入）未在架构层显式建模。
4. **迭代与首轮流程割裂**：首轮（fresh_start）走完整链路，后续迭代（incremental）只走验证环节——两者编排正交性差，缺乏统一的迭代抽象。

### 1.2 期望的新架构

- 提供**多模式迭代入口**：支持全量迭代 / 前端单侧迭代 / 后端单侧迭代 / 文档层迭代等多种模式，由迭代入口角色基于改动范围路由。
- **迭代反馈回路分层**：关键节点（前端实现、后端实现）支持快速局部回退，不必每次走全栈联调。
- **增量基线显式化**：迭代轮次之间的产出物传递应在架构层有明确路径（carries 边）。
- **保留前后端分离主线**（约束条件，见 §3）。

## 2. 角色（目标架构预期涉及）

目标架构为 lxp-eng-planning 重新设计后的多角色协作系统，预期涉及以下角色类型：

| 角色类型 | 示例 | 职责 |
|---------|------|------|
| **producer（入口）** | 需求接收者 / 迭代编排者 | 接收用户需求或迭代指令，产出需求文档或迭代策略 |
| **standard（分流/设计）** | 需求分流者 / 前后端设计组 | 基于需求分流到前后端设计链路 |
| **producer（实现）** | 前端实现者 / 后端实现者 | 产出前端/后端代码 |
| **standard（校验/对抗）** | 交叉审核者 / 合并校验者 / 综合裁决者 | 三层对抗治理 |
| **standard（迭代专属，新增/重构）** | 迭代策略决策者 | 替代当前"迭代守门人"，支持多模式路由 |

**至少 2 个角色（含 1 个 producer 入口）**——目标架构预期保留 ≥10 个角色，其中 producer ≥3 个（需求接收者 + 前端实现者 + 后端实现者）。

## 3. 边界（做什么 / 不做什么）

### 3.1 做什么

1. **重新设计 lxp-eng-planning 的角色编排拓扑**（app.yaml 中的 roles + edges）。
2. **重构迭代入口机制**：替代当前的"迭代守门人 + 三选一"，设计支持多模式迭代的新入口。
3. **优化迭代反馈回路**：为关键节点设计局部快速回退边，缩短试错路径。
4. **显式建模增量基线**：迭代轮次间的产出物传递通过 carries 边清晰表达。
5. **产出架构蓝图**：含 7 个强制章节（系统总体 / 层划分 / 模块清单+RACI / 并行兼容性 / 文档层级 / 失败模式与应对 / 对抗记录）。

### 3.2 不做什么

1. **不改变 lxp-eng-planning 的业务领域**（精力管理 / 工程规划）。
2. **不重新设计具体角色的 skill 内容**（只设计架构，不写角色执行逻辑）。
3. **不删除前后端分离主线**（约束条件）——前端实现者、后端实现者必须保留。
4. **不修改现有的 knowledge 知识文档**（knowledge/ 目录内容不动）。
5. **不产出前端/后端代码**（pure-arch-design 只产架构蓝图 Markdown）。
6. **不破坏三层对抗治理结构**（交叉审核 / 合并校验 / 综合裁决的核心定位保留，具体拓扑可优化）。

## 5. 现状参考（供需求分析师和架构设计师参考）

### 5.1 当前 lxp-eng-planning 架构快照

- **路径**：`/Users/xiaopenglv/ai-agents/多角色配合系统/apps/lxp-eng-planning/app.yaml`（377 行）
- **角色数**：18（含 4 个 manual 节点：迭代守门人 / 需求接收者 / 全栈联调验证者 / 综合裁决者）

| 当前 | 期望 |
|------|------|
| 迭代守门人三选一（fresh_start/incremental/done） | 多模式迭代入口（全量/前端单侧/后端单侧/文档层等） |
| incremental 跳到全栈联调验证者 | 增量基线显式建模，按改动范围路由到对应链路起点 |
